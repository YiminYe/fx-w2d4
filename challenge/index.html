Today's tasks

1 · The three altitudes

AM · ~90 min · deck: Testing / JUnit · Mockito · Testcontainers

Why: "the build is green" means nothing until you know what each test can and can't catch.

Three tests, three heights, all in com.fx.api. A unit test (ConversionServiceTest) mocks RateRepository and proves the maths in isolation. A slice test (RateControllerTest, @WebMvcTest + @MockBean) proves the HTTP contract without the full app. An integration test (RateRepositoryIT, Testcontainers) runs your real SQL against a throwaway MySQL that Liquibase seeds on startup — no seed file, no local database. Then the pom split: <testcontainers.version>1.21.4</testcontainers.version> and maven-failsafe, so *Test and *IT run in different phases.

Stuck? Boot 3.3.4 manages Testcontainers 1.19.8, which 400s on Docker 28/29+. With the wrong version mvn verify reports BUILD SUCCESS while the IT is silently skipped — a green build that tested nothing. Pin 1.21.4.
Checkpoint: mvn test → 29 run, 0 fail, 1 skipped (a pre-existing @Disabled, not the IT). mvn verify → 29 + 3 IT green. Unit: 123.45 EUR→USD @1.0818 → converted 133.55, fee 1.34, net 132.21. IT: findLatest() size 10, EUR/USD 1.0818.

2 · The machine runs the tests (CI)

AM/PM · ~45 min · deck: CI / GitHub Actions

Why: a test only protects you if it runs without you remembering to run it.

Author .github/workflows/ci.yml at the repo root (working-directory: fx-app-spring), triggered on push and pull_request. Three jobs: build (mvn -B verify), lint (mvn -B validate), and dependency-check (OWASP, continue-on-error for now). Then rig a PR with a deliberately failing test, watch CI go red, and turn on branch protection so red can't merge.

Stuck? "Required status checks" in branch protection is what turns a red X from a suggestion into a locked merge button. No protection rule → nothing is actually enforced.
Checkpoint: a green push shows build · lint · dependency-check; the rigged red PR is blocked by branch protection — the merge button won't go.

3 · Supply chain & the Definition of Done

PM · ~45 min · deck: Supply chain / dependencies

Why: most of your app is code you didn't write — and every line of it is your attack surface.

Read mvn dependency:tree and find something you never chose to depend on. Look up a real CVE. Turn on Dependabot and wire the OWASP dependency-check job. Then the honesty drill: stop Docker, run mvn test, watch it say BUILD SUCCESS with the IT skipped — and prove you can tell it skipped. Write it all up in docs/day4-status.md with the team's five-point Definition of Done.

Stuck? A skipped test is not a passed test. In the surefire output, "Tests run: … Skipped: 1" for the IT is the tell — a green summary line is not the same as coverage.
Checkpoint: Docker off → mvn test is BUILD SUCCESS but RateRepositoryIT is skipped, and you caught it. docs/day4-status.md carries the 5-point DoD.

4 · From green build to blessed deploy (CD)

PM · ~45 min · deck: CI→CD

Why: shipping should be one click a human takes on purpose — not a thing that happens by accident.

Split the pipeline by trigger so nothing builds twice: narrow ci.yml to pull_request, then add cd.yml on push: [main]. Its jobs chain: build (verify) → docker (log in to ghcr.io, build, push sha-<short> and latest to ghcr.io/<owner>/fx-app-spring) → deploy (environment: staging, a required-reviewer gate; the deploy itself is a stand-in echo for now).

Stuck? ghcr package names must be lowercase. If github.repository_owner has capitals, the push 400s — lowercase it in the workflow.
Checkpoint: a merge to main runs cd.yml; the fx-app-spring package on ghcr shows two tags — sha-<short> and latest; the deploy job waits for a reviewer before it runs.

Theory background

Three altitudes, one pyramid

Many fast unit tests at the base (mock the collaborators, prove one thing), fewer slice tests in the middle (one layer, faked below it), a couple of slow integration tests at the top (the real thing, end to end). Each altitude catches what the one below it can't — and none of them, alone, tells you the system works.

mvn test is not mvn verify

Surefire runs in the test phase and executes classes named *Test — your unit and slice tests. Failsafe runs later, in integration-test/verify, and executes *IT — your integration test. The split is a naming convention, nothing more. mvn test is fast and Docker-free; mvn verify is the honest one.

The green build that tested nothing

Boot 3.3.4 manages Testcontainers 1.19.8, which fails on Docker 28/29+ with an HTTP 400 "could not find a valid Docker environment". The trap: instead of failing, the IT is silently skipped and the build stays green. A build that reports success while testing nothing is worse than a red one, because it lies. The one-line fix — <testcontainers.version>1.21.4</testcontainers.version> — is the whole point of the morning.

CI is a gate, not a chore

A pipeline that runs the suite on every push tells you when something broke. Branch protection — a required status check — is what turns that from information into enforcement: a red run doesn't just warn you, it locks the merge button. Green stops being an opinion and becomes a precondition.

CD ships; a human blesses

The CD pipeline builds the image, tags it sha-<short> so every deploy is traceable to an exact commit, and pushes it to the registry — all automatic. The deploy job then sits behind environment: staging with a required reviewer. Automation carries the change to the door; a person decides to open it. That gate is the difference between automation and abdication.

Your dependencies are your attack surface

Most of your running bytes are libraries you never read. mvn dependency:tree shows you what you actually pulled in — including the transitive things you never chose. Dependabot watches them for known CVEs and opens PRs; the OWASP dependency-check job fails the build when one crosses your threshold. You can't audit every line, but you can refuse to ship a known hole.

Extra challenges

Test the feed endpoint

Yesterday's POST /api/feed/rates writes to the database and has no test at all. Add a slice test for the happy path (202) and a malformed batch (4xx), then an *IT that proves the rows actually land. Which altitude belongs to an endpoint whose whole job is a side effect?

Fail CI on a real CVE

The OWASP job is continue-on-error: true — it reports but never blocks. Pin a dependency with a known vulnerability (or drop the CVSS fail-threshold), watch the job turn red, then remove continue-on-error and feel the gate actually bite. When should a CVE block a merge, and when is it noise?

Add a smoke-test job that curls the running container

After cd.yml pushes the image, add a job that docker run -d that exact image, waits for /health, curls /api/rates and asserts it got 10, then tears it down. A green unit suite and a container that won't start are not mutually exclusive — this catches the gap.

Sign the image / attach an SBOM

Use cosign to sign the ghcr image, or syft to generate an SBOM and attach it to the package, then verify it from a fresh checkout. "Where did this image come from and what's inside it?" is the next supply-chain question after "is it vulnerable?"
