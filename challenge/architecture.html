The tests,
and the pipeline that runs them

Two pictures. First, the shape of a healthy test suite — a wide base of fast unit tests, a couple of slow, real ones at the top. Then the pipeline that runs all of it on every change, and gates both the merge and the deploy.

fast, mocked — a unit test / a green build a slice test / a CI job / the app slow & real — the IT / a human gate a database a gate you must pass

The same green build flows through both diagrams. A test only protects you once a pipeline runs it without you remembering to — and a deploy is only safe once a human still has to open the last door.

The test pyramid — three altitudes

Task 1 · com.fx.api
What each is for: unit proves the maths with the repository mocked; the slice proves the HTTP contract with the repository a @MockBean; the IT proves the real SQL against a throwaway MySQL that Liquibase seeds on startup — no seed file. Nine new tests across the three classes; the whole suite is 29 run + 3 IT.
Watch for: only the IT needs Docker. With the version Boot 3.3.4 ships (1.19.8), mvn verify prints BUILD SUCCESS while the IT is silently skipped — coverage you think you have and don't. Pin <testcontainers.version>1.21.4</testcontainers.version>.

The pipeline that runs them — CI gates the merge, CD ships

Tasks 2 & 4 · at the repo root
Split by trigger, not duplicated: ci.yml runs on pull_request and gates the merge; cd.yml runs on push: [main] and ships what merged. Both live at the repo root with working-directory: fx-app-spring. No commit gets built twice.
Watch for: the push lands the image at ghcr.io/<owner>/fx-app-spring tagged sha-<short> and latest — the owner MUST be lowercase or it 400s. Then deploy stops at environment: staging for a required reviewer: automation up to the door, a human opens it.
